## Problem

Pull requests can introduce architecture drift when code does not match repository structure, ADRs, or conventions. A reviewer may not immediately know whether a change fits the existing architecture, especially in a repository with multiple services, conventions, and historical decisions.

## Why It Matters

Teams need early feedback on architectural fit before merging code. ArchGuard focuses on making that context easier to inspect by turning repository signals into a clear GitHub Check Run verdict before review debt piles up.

## What I Built

Built a GitHub App integration that receives pull request webhooks, analyzes repository context, and posts GitHub Check Run verdicts for architecture fit. The project includes a public demo UI, a live API/worker deployment, and proof endpoints for recruiter review.

## Architecture / Workflow

GitHub PR webhook → Fastify API → Redis/BullMQ worker → repository context retrieval → RAG-style analysis → GitHub Check Run verdict.

1. A pull request event arrives through GitHub webhooks.
2. The Fastify API validates and queues the review job.
3. Redis and BullMQ move work into a background worker pipeline.
4. Prisma and PostgreSQL persist repository, pull request, and review state.
5. Repository context, ADRs, and codebase conventions are retrieved for RAG-style analysis.
6. The result is posted back to GitHub through the Checks API as a review verdict.

## Verdicts

- `FIT`
- `DRIFT_RISK`
- `INSUFFICIENT_EVIDENCE`
